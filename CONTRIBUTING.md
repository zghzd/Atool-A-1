# 关于
本文章将指导你进行issues和Pull requests提交，
文章给你提供一些建议，这有助于更好地处理你的消息，你可以
选择性的接受文章建议。  
**允许自行对给定的格式进行扩展，以便其符合实际。**
# issues
项目将issues分为两种，一种是对已存在问题的提交（bug等），
另一种是对未来版本的建议（新功能请求等）。这两种内容采用
不同的格式。
## 已存在问题提交
```
---
简短的问题描述。
---
复现步骤1.
    2.
    3.
    ...
---
预期结果，以及复现结果
---
信息：
    操作系统及操作系统版本：
    CPU架构：
    架构转译设备/软件（如有，一般在非amd64架构上）信息：
    复现问题的Atool-A-1版本（及可选的git提交编号）：
    复现问题的时间（附带时区）：
---
同时，附加以下文件：
    复现时的日志文件
---
```
## 新功能请求
```
---
标题
---
需求内容
---
理由
---
实际建议内容（伪代码、代码、数据等）（可选，但必须标注原创）
---
```
# Pull requests
Pull requests，简称PR，如果你完成了项目修改，可以通过提交PR
来请求合并你的更改。
通常，这需要一些时间用于审核，你需要耐心等待。
在PR中，你需要明确的点出：
- 你修改了什么
- 修改的原因
- 可能产生的影响
- 修改人（作者）
